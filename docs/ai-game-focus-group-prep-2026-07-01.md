# 《心屿》AI 游戏焦点小组访谈准备稿

> 访谈场景：中国传媒大学数字人研究院 × 腾讯研究院 AI 游戏报告线下深度焦点小组访谈  
> 时间：2026-07-01 20:00，约 1 小时  
> 项目：心屿 Xinyu  
> 目标：把《心屿》作为一个可被写进行业报告的 AI 原生游戏案例讲清楚。  

## 0. 一句话总策略

《心屿》不要被讲成“AI 聊天应用”，而要被讲成：

> **一款 AI 驱动的情绪世界生成游戏：AI 理解用户情绪，代码把它转成稳定的游戏状态，3D 岛屿把这种状态变成可探索、可回访、可留下痕迹的世界。**

访谈里的主线始终围绕四句话：

1. **AI 不是只生成文本，而是参与世界状态生成。**
2. **CodeBuddy 和腾讯混元让小团队完成了过去多岗位协作才能完成的创作闭环。**
3. **情感类 AI 游戏的关键不是“更像人”，而是更克制、更可解释、更安全。**
4. **《心屿》的价值在于把情绪从一次性表达，变成长期可回访的游戏记忆。**

## 1. 30 秒、2 分钟、5 分钟三个版本

### 1.1 30 秒版本

《心屿》是一款 AI 驱动的情绪世界生成游戏。用户输入一句心情，甚至只写一个字或静默坐一会儿，系统会理解情绪，生成克制的中文叙事，并把这一刻沉淀为岛屿记忆、物件、场景变化和 3D 探索内容。我们使用 CodeBuddy 辅助工程开发和测试迭代，也使用腾讯混元模型参与中文叙事、角色设定和 3D 建模方案生成，再通过 Blender 与 three.js 接入成可运行体验。

### 1.2 2 分钟版本

《心屿》的出发点是：很多人有焦虑、疲惫、孤独和压力，但并不一定愿意立刻找人倾诉，也不一定能组织出完整的话。所以我们做了一座“会回应你的心象岛屿”。用户可以输入一段话，也可以写一个心境字，甚至什么都不说，只在岛上坐一会儿。

系统会把这次输入走一条可解释链路：情绪分析、记忆检索、环境推理、叙事表达、安全边界。AI 负责理解情绪和生成叙事；安全规则、岛屿成长、记忆存储、3D 降级都由确定性代码负责。最后用户看到的不是一段聊天回复，而是一座岛发生了变化：天气、音乐、灯塔、花、星光、物件、记忆树都会跟着情绪和历史改变。

在创作工作流上，我们用 CodeBuddy 做 AI 工程协作，包括需求拆解、前后端实现、3D 资产接入、测试和文档；用腾讯混元模型辅助中文语境下的情绪叙事、岛屿意象、角色与物件设定，以及 3D 建模提示和方案推演。最终资产通过 Blender 脚本导出为 GLB，在 react-three-fiber / three.js 中渲染成可漫游的心象岛屿。

### 1.3 5 分钟版本

《心屿》不是传统意义上的功能工具，也不是只加了聊天框的 AI 游戏。它尝试回答一个问题：**如果 AI 坐进开发者的工位，它不只是写对白，而是参与策划、叙事、美术资产、工程和验证，那么一个小团队能否做出一个完整的 AI 原生游戏闭环？**

我们的答案是“可以，但必须把 AI 放在有边界的系统里”。《心屿》的核心输入是用户情绪，核心反馈是世界变化。用户说“我今天很累，感觉什么都做不好”，系统不会只回一句“你辛苦了”，而是把这句话拆成情绪、强度、场景、记忆和岛屿状态：岛屿可能进入夜色，出现星光、吊床、灯塔；系统生成一段克制的叙事和一句心灵印记；这条记录会进入记忆库，未来回访时岛屿可以说“我记得你上次也带着疲惫来过”。

项目里最重要的设计取舍是：**该用 AI 的地方用 AI，该用规则的地方用规则。** 情绪分析和叙事表达适合用大模型；岛屿成长等级、趋势判断、安全触发、隐私删除、WebSocket 回退、3D 降级更适合写成确定性代码。尤其是安全边界，我们没有把它完全交给模型，而是做了情绪强度阈值、关键词黑名单、文本归一化防绕过和输出端复检。

3D 方面，我们把 AI 创作链路落到了实际资产：角色、NPC、记忆树、心灵印记、星愿、静默贝壳、心境石、钓鱼浮标、流星、村落和海岸道具等，最终都通过 GLB 进入 Web 端 3D 场景。这里 CodeBuddy 主要提高工程闭环速度，腾讯混元主要提高中文创意与 3D 资产设定效率，Blender 负责模型整理与导出，three.js 负责实时渲染、动画、交互和性能分级。

所以我希望在报告里表达的核心观点是：**AI 游戏不应该停在“模型能生成内容”，更重要的是模型生成的内容能否进入稳定的游戏规则、长期记忆和可验证的体验。**

## 3. 从源码解析项目架构

### 3.1 总体架构

源码体现出《心屿》不是单页 Demo，而是一个完整 Web 游戏应用：

| 层 | 主要文件 | 作用 |
|---|---|---|
| 前端主体验 | `frontend/src/pages/Home.tsx` | 首页、输入、流式反馈、叙事、安全、3D 入口 |
| 3D 自由探索 | `frontend/src/components/ExploreMode.tsx` | 可漫游岛屿、角色、模型、精灵、玩法、性能分级 |
| 前端 API | `frontend/src/lib/api.ts` | HTTP / WebSocket / TTS / 对话接口封装 |
| 后端入口 | `backend/app/main.py` | FastAPI 路由、反思管线、TTS、对话、回访、隐私删除 |
| AI Provider | `backend/app/services/llm_provider.py` | Mock 与 OpenAI 兼容模型抽象，可指向腾讯混元 |
| 工具型 Agent | `backend/app/services/agent_service.py` | function-calling 工具循环：记忆检索、读岛屿 |
| 安全规则 | `backend/app/services/safety_service.py` | 高风险输入/历史记忆检测，纯规则 |
| 岛屿状态 | `backend/app/services/island_state_service.py` | 1-5 级成长、趋势、元素、章节导语 |
| 场景映射 | `backend/app/services/scene_map.py` | 8 情绪 × 3 强度 = 24 个场景 |
| 记忆系统 | `backend/app/services/memory_service.py` + `vector_memory_service.py` | PostgreSQL 事实源 + pgvector 语义检索 |
| 3D 模型脚本 | `blender/*.py` | Blender 离线生成或整理 GLB |

### 3.2 后端反思主链路

源码里的核心函数是 `backend/app/main.py` 的 `_classic_reflect_pipeline`、`_agent_reflect_pipeline` 和 `_reflect_pipeline`。

它有两条路径：

| 路径 | 触发条件 | 特点 |
|---|---|---|
| 经典反思管线 | 默认、Mock 模式、高风险关键词命中、模型不可用 | 稳定、可解释、适合演示 |
| 工具型 Agent 管线 | 配置真实 OpenAI 兼容模型且无高风险关键词 | 模型可自主调用 `recall_memories` / `read_island` |

经典管线顺序：

1. 读取最近记忆。
2. `emotion_service.analyze` 判断情绪、强度、摘要。
3. `safety_service.check` 前置判断高风险。
4. `vector_memory_service.search` 尝试语义检索历史记忆，失败回退最近记忆。
5. `scene_map.get_scene` 把情绪映射到场景。
6. `island_state_service.compute` 计算岛屿成长、趋势和元素。
7. 安全未触发时调用 `narrative_service.generate` 生成叙事和心灵印记。
8. `_scrub_generated` 对模型输出做二次安全复检。
9. 非无痕模式保存记忆，并尝试写入向量索引。
10. 返回 `ReflectResponse`，前端展示叙事、导演台、选择卡和岛屿状态。

现场讲法：

> 我们把一次情绪输入拆成一条可解释管线。AI 只负责最适合它的两步：理解和表达；安全、成长、记忆、回退都由代码控制。这样既有 AI 的温度，也有游戏系统的稳定性。

### 3.3 WebSocket 流式与幂等

`frontend/src/lib/api.ts` 使用 `reflectStream` 先走 `/ws/reflect`；失败后 `Home.tsx` 回退 `POST /api/reflect`。前端每次提交生成一个 `request_id`，后端 `_run_reflect_idempotent` 和 `_reflect_cache` 用这个 ID 防止 WebSocket 超时后 HTTP 回退造成重复落库。

可讲价值：

> 情绪陪伴体验最怕用户等一个黑箱。我们用 WebSocket 分阶段推送，让“潮汐、贝壳、岛屿天气、羽毛笔、灯塔”逐个抵达。即使网络失败，也用同一个 request_id 回退 HTTP，不重复写记忆。

### 3.4 安全边界

`SafetyService` 是纯规则，不调用模型。它做了几层事：

1. 高风险负面情绪 + 强度达到阈值触发。
2. 中英双语关键词黑名单触发。
3. 文本归一化：全角转半角、去空格标点、统一小写，防止“想 死”“自。杀”这种绕过。
4. 历史记忆复检：`memory_is_high_risk` 防止回访、年报、低语复述高风险内容。
5. 输出端 `_scrub_generated` 防止模型生成文本意外触发风险词。

现场讲法：

> 安全不是模型“自觉”，而是系统规则。情感类 AI 游戏不能把危机处置交给模型临场发挥。我们做了输入端和输出端双保险，高风险时停止普通叙事，不给选择卡，引导用户联系可信任的人或专业资源。

### 3.5 岛屿成长状态机

`IslandStateService` 是纯 Python，零 LLM。

| 机制 | 规则 |
|---|---|
| 成长等级 | 记忆数量决定 1-5 级 |
| 主导情绪 | 最近记忆中出现最多，平局优先更新 |
| 趋势 | `recovering`、`brightening`、`stormy`、`stable`、`mixed` |
| 元素 | 情绪元素 + 玩家物件合并，最多展示 8 个 |
| 章节导语 | 描述岛屿状态，不预测、不承诺疗效 |

可讲价值：

> 岛屿不是每次随机生成，而是一个稳定状态机。AI 的输入会进入状态机，但世界怎么长由规则决定。这样玩家回访时才有连续感。

### 3.6 场景映射

`scene_map.py` 明确有 8 种情绪 × 3 个强度档 = 24 个场景：

| 情绪 | 低/中/高强度视觉方向 |
|---|---|
| `sad` | 黄昏小雨 → 黄昏雨浪 → 夜雨远光 |
| `anxious` | 晨雾 → 雾锁礁石 → 高压风暴 |
| `tired` | 傍晚吊床 → 夜晚星光 → 星寂深夜 |
| `lonely` | 黎明雾树 → 淡雾海岸 → 月光空岸 |
| `calm` | 晨光海面 → 白日小帆 → 黄昏琉璃潮 |
| `happy` | 清晨暖光 → 花与蝴蝶 → 金色夕照 |
| `angry` | 绯风 → 风暴 → 黑夜雷云 |
| `helpless` | 微雨暗岸 → 夜雨静水 → 微光残烬 |

可讲价值：

> 用户不是只得到一段回复，而是看到情绪被翻译成时间、天气、颜色、音乐和意象。

### 3.7 记忆系统

源码里记忆分两层：

| 层 | 作用 |
|---|---|
| PostgreSQL `memories` | 事实源，保存用户文本、情绪、强度、叙事、印记 |
| pgvector `memory_vectors` | 增强检索，用语义找相关旧记忆 |

`VectorMemoryService` 任何失败都会返回空，主流程回退最近记忆，不阻断体验。`delete_identity` 会删除记忆、物件、私房话和向量。

可讲价值：

> 我们把“它记得我”做成真实的数据结构，而不是一句假装记得的话。同时用户可以删除全部后端痕迹。

## 4. 前端体验解析

### 4.1 输入入口

`MoodInput.tsx` 支持：

1. 文本输入。
2. 浏览器语音输入。
3. 快捷短句。
4. 无痕模式。
5. 静默坐岛入口。
6. 写一个字入口。

其中“无痕模式”的现场表达很重要：

> 我们承认用户有时候只想被陪一下，不想被记录。无痕模式仍然让岛屿回应你，但不写记忆、不留物件、不推进成长。

### 4.2 五感信使导演台

`AgentDirectorPanel.tsx` 把后端 `agent_trace` 映射成五位信使：

| 信使 | 对应后端阶段 | 用户感知 |
|---|---|---|
| 潮汐 | emotion | 替你读懂心情起落 |
| 贝壳 | memory | 翻找岛屿记得的你 |
| 岛屿天气 | environment | 为这一刻挑一片天 |
| 羽毛笔 | narrative | 把回应写在风里 |
| 灯塔 | safety | 守住最深的那段海 |

现场讲法：

> 我们把 AI 过程叙事化，不让用户面对一个黑箱。用户可以看到系统如何一步步回应自己。

### 4.3 呼吸仪式

`Home.tsx` 有 `shouldOfferBreathing`：高强度负面情绪但未触发安全硬阻断时，会进入呼吸仪式。这说明项目不是只生成文本，还把情绪强度转成交互节奏。

现场讲法：

> 如果情绪太满，但不属于危机表达，我们不会急着给叙事，而是先邀请做一个短暂的着陆动作。

### 4.4 3D 入口与性能

前端首页里 `Island3D` 和 `ExploreMode` 都是 `lazy` 懒加载。`prefetchExploreAssets` 会在桌面端空闲或靠近按钮时预热模型，但移动端/触屏设备跳过后台预热，避免卡顿。

可讲价值：

> 我们没有为了展示 3D 牺牲可用性。弱设备、无 WebGL、低性能都会降级，不让用户白屏。

## 5. 3D 自由探索解析

### 5.1 3D 资产系统

`ExploreMode.tsx` 的 `MODELS` 表接入了大量 GLB：

| 类别 | 示例 |
|---|---|
| 地标 | 灯塔、风车、鸟居、神社、码头、船、鲸石 |
| 角色 | 新主角 `xyshz_rigged.glb`、守护者、Pocoyo、捏人 avatar、村民 |
| 记忆物件 | 心灵印记、记忆树、星愿、静默贝壳、心境石 |
| 自然 | 松树、阔叶树、灌木、花、岩石、莲、芦苇、作物 |
| 海滩 | 海星、漂木、沙堡、珊瑚、躺椅、冲浪板、螃蟹、海龟、宝箱 |
| 玩法 | 天灯、钓鱼浮标、风铃、汽车 |

现场讲法：

> 我们不是只做了一张 3D 背景图，而是把记忆、物件、NPC、地标、玩法都接进同一个可漫游世界。

### 5.2 角色与动作

源码显示可切换四种主角：

1. `hero`：新默认心屿守护者，使用 `xyshz_rigged.glb`。
2. `guardian`：旧的记忆守护者。
3. `pocoyo`：修正轴向和落脚点的 GLB。
4. `avatar`：用户捏的人。

`xyshz` 角色有完整动作库：`Idle`、`WalkLoop`、`RunLoop`、`Jump`、`Wave`、`Flute`、`Sit`、`Cheer`。测试文件验证了动作接线、转向、跑步、跳跃、过渡、身体起伏等逻辑。

现场讲法：

> AI 生成或辅助生成的角色，只有进入动作系统、输入系统和测试系统，才真正从“模型”变成“游戏角色”。

### 5.3 陪伴精灵“微光”

精灵系统由三部分构成：

| 层 | 文件 | 能力 |
|---|---|---|
| 3D 表现 | `ExploreMode.tsx` | 跟随玩家、灯光、口型、唱歌、睡觉、头顶气泡 |
| 本地养成 | `companionSpirit.ts` | 亲密度、投喂、谈心、秘密解锁 |
| AI 对话与语音 | `/api/companion/chat` + `companionVoice.ts` | 后端生成回复，云端 TTS 优先，浏览器语音降级 |

精灵不是普通 NPC，它有：

1. 名字可改。
2. 亲密度 0-100。
3. 投喂次数、对话次数。
4. 唱歌、合唱、哄睡、深夜相伴。
5. 主动陪聊开关。
6. 云端音色选择。
7. 说话时 3D 口型和灯塔光脉动。

现场讲法：

> 精灵“微光”是 AI 角色进入 3D 空间的实验。它不是站在 UI 里的聊天框，而是跟着你走、会开口、会发光、会通过养成状态记住你们之间的关系。

### 5.4 自由探索玩法

源码里有多种轻玩法：

| 玩法 | 说明 |
|---|---|
| 心灵印记 | 历史记忆变成 3D 可拾取印记 |
| 记忆树 | 拾完印记后长出记忆树，颜色来自已拾取印记 |
| 放天灯 | 写下心事，天灯升空，支持单盏和一片 |
| 垂钓 | 抛竿、等待、咬钩、节奏收线 |
| 风铃心曲 | 按顺序敲响风铃，完成后满岛萤火 |
| 种花 | 在岛上留下花 |
| 开车 | 可在岛上开，也可进入林间土路驾驶场景 |
| 奇遇 | 鲸落、漂流瓶、螃蟹、海龟、宝箱、海螺等 |
| 写信 | 邮筒、给未来自己的瓶中信 |

现场讲法：

> 这些玩法都很轻，但它们让《心屿》从“生成叙事”变成“可玩空间”。玩家可以把情绪放下、带走、种下、点亮、捞起。

## 6. CodeBuddy 与腾讯混元的完整讲法

### 6.1 不要说“AI 一键做完”

不建议说：

> “这个项目都是 AI 自动生成的。”

建议说：

> 我们采用的是 AI 辅助创作工作流。CodeBuddy 像工程搭档，腾讯混元像创意和建模设定助手，人负责判断、取舍、安全边界、审美和验证。

### 6.2 CodeBuddy 的价值

CodeBuddy 在项目里可以这样讲：

| 阶段 | CodeBuddy 作用 |
|---|---|
| 需求拆解 | 把“情绪岛屿”拆成输入、叙事、记忆、岛屿状态、3D、精灵、安全 |
| 后端实现 | FastAPI 路由、服务层、TTS、对话、回访、删除、幂等 |
| 前端实现 | React 组件、状态流、WebSocket、3D 入口、移动端适配 |
| 3D 接入 | GLB registry、模型缩放、材质、角色动作、道具摆放 |
| 测试 | 后端回归测试、前端模型测试、3D 节点/材质名测试 |
| 文档 | README、需求文档、视觉检查、建模 backlog |

一句话：

> CodeBuddy 最大价值不是写某一段代码，而是让复杂项目的跨模块迭代能快速闭环。

### 6.3 腾讯混元的价值

腾讯混元可以从三个方向讲：

| 方向 | 价值 |
|---|---|
| 中文情绪叙事 | 更适合中文语境下的克制、温柔、意象化表达 |
| 世界观与物件设定 | 灯塔、贝壳、心境石、记忆树、精灵、天灯等意象生成 |
| 3D 建模方案 | 角色/物件结构、材质、部件命名、风格提示、建模说明 |

推荐完整口径：

> 我们在项目中把腾讯混元视为中文创意和 3D 资产设定的协作者。它可以帮助生成角色设定、岛屿物件、心灵印记、记忆树、静默贝壳、心境石等模型方案，再由 Blender 脚本整理或生成资产，导出 GLB 后接入 three.js。最终上线的体验仍要经过工程接入、视觉检查、性能优化和测试验证。

### 6.4 AI 参与 3D 建模链路

| 步骤 | 内容 | 产物 |
|---|---|---|
| 1. 玩法需求 | 需要“情绪记忆能被看见和拾取” | 心灵印记、记忆树 |
| 2. 语义设定 | 用混元生成物件意象、风格、部件说明 | 星、贝壳、花、火花、水滴 |
| 3. 模型规划 | 确定节点名、材质名、轮廓、缩放 | GLB 合同 |
| 4. Blender 脚本 | 生成/整理模型，导出 GLB | `frontend/public/models/*.glb` |
| 5. 前端 registry | 声明模型 URL、scale、tag、emotion mapping | `artifact3d.ts` / `imprint3d.ts` |
| 6. 场景接入 | `ExploreMode` 中渲染、拾取、动画 | 可玩对象 |
| 7. 自动测试 | 文件存在、节点名、材质名、路由 | `npm test` |
| 8. 视觉检查 | Playwright 截图确认比例和可读性 | `docs/*visual-inspection.md` |

## 9. 一小时访谈问题库

### 9.1 你怎么看“AI 坐进开发者工位”？

回答：

> 我觉得 AI 坐进开发者工位之后，最先改变的是原型速度和岗位边界。以前一个情绪陪伴 3D 游戏需要策划、文案、前端、后端、美术、测试多个角色并行，现在 CodeBuddy 和混元能把很多中间环节压缩。  
> 但 AI 并没有替代人的判断。产品边界、安全、审美、系统架构、是否真的好玩，仍然需要人来负责。我们的经验是：AI 越强，越需要工程系统把它约束住，让它输出的内容能进入稳定规则，而不是散落成一堆漂亮但不可控的素材。

### 9.2 《心屿》和普通 AI 聊天有什么不同？

回答：

> 普通 AI 聊天的输出主要是一段回复。《心屿》的输出是一座世界：情绪会影响天气、音乐、3D 场景、记忆、物件、回访和角色互动。聊天只是入口，游戏状态才是结果。

### 9.3 AI 在心屿里承担什么角色？

回答：

> AI 有三种角色。第一是理解者，判断情绪和强度；第二是叙事者，生成克制的岛屿语言；第三是创作协作者，参与角色、物件、3D 建模设定和工程实现。但安全和状态机由代码负责，这样体验可控。

### 9.4 为什么做情绪陪伴题材？

回答：

> 因为很多人不是缺一个效率工具，而是缺一个低压力、不会评价自己、可以随时离开的空间。游戏比工具更适合承载这种陪伴，因为游戏可以允许沉默、允许绕路、允许把情绪变成一个可以慢慢看的场景。

### 9.5 为什么一定要做 3D？

回答：

> 文字能表达“我听见你了”，但 3D 世界能表达“你来过”。当用户看到自己的心灵印记、记忆树、灯塔、天灯和花留在岛上，情绪就不再只是一次输入，而是变成了空间里的痕迹。

### 9.6 最大挑战是什么？

回答：

> 最大挑战不是让 AI 多说，而是让 AI 少说错、说得克制，并且让它的输出能稳定进入游戏系统。情感陪伴尤其不能炫技，不能过度承诺，也不能把危机内容交给模型自由发挥。

### 9.7 CodeBuddy 最具体帮了什么？

回答：

> CodeBuddy 帮我们把复杂跨模块迭代变快了。例如一个新 3D 物件，不只是做模型，还要注册 URL、接入组件、摆放位置、控制缩放、写测试、做视觉检查、更新文档。CodeBuddy 能帮助我们在这些环节之间来回穿梭，减少遗漏。

### 9.8 腾讯混元最具体帮了什么？

回答：

> 混元对中文语境和意象生成很有帮助。像“心灵印记”“静默贝壳”“心境石”“记忆树”“微光精灵”这些不是单纯功能命名，而是需要中文审美和情绪语气。混元可以帮助我们生成设定、风格和建模提示，再由工程链路落地为 GLB 资产和 3D 交互。

### 9.9 你们如何看 AI 游戏的未来？

回答：

> 我认为 AI 游戏下一步不只是 NPC 更会聊天，而是游戏世界更会响应玩家。AI 应该进入任务生成、世界状态、角色关系、个性化资产和长期记忆。但越是这样，越要有边界、验证和可解释性。

## 10. 不要踩的坑

| 不建议说 | 改成 |
|---|---|
| AI 治疗用户 | AI 情感陪伴，不提供心理咨询或医疗服务 |
| 全部由 AI 自动生成 | AI 辅助创作，人工负责判断、边界和验证 |
| 五个 Agent 完全自治 | 可解释的情感 AI 编排管道；对话链路才是工具型 agent |
| 模型完全可靠 | 模型输出受规则、安全网和降级机制约束 |
| 心屿保证用户变好 | 心屿不承诺疗效，只提供陪伴和回望 |
| 3D 都是模型一键生成 | 混元辅助设定，Blender 整理/生成，工程接入与测试验证 |

## 11. 可引用金句

1. **AI 游戏的关键不是生成更多内容，而是让生成内容进入稳定的游戏规则和世界状态。**
2. **《心屿》不是让 AI 安慰玩家，而是让 AI 把玩家的情绪变成一座会生长的世界。**
3. **CodeBuddy 和腾讯混元让小团队拥有了更完整的创作链路，但人的判断仍然是产品边界。**
4. **情感类 AI 游戏最重要的不是更像人，而是更克制、更可解释、更安全。**
5. **聊天只是入口，岛屿变化才是结果。**
6. **我们希望用户感到：我来过，我被记得，我的情绪在这个世界里留下了形状。**
7. **AI 负责理解和生成，代码负责边界和验证，玩家感受负责最终裁判。**

## 13. 最后收束稿

访谈快结束时可以这样收束：

> 如果要总结《心屿》的经验，我会说它不是一个“AI 加游戏”的简单组合，而是试图把 AI 放进游戏创作的多个环节：它理解玩家情绪，也参与叙事、美术设定、3D 建模方案和工程实现。但我们同时把安全、隐私、成长状态、降级和验证牢牢放在代码里。  
> 对我们来说，AI 原生游戏不是模型更会说话，而是玩家每一次输入都能改变世界，并且这种改变是可回访、可解释、可删除、可验证的。
