# Companion Spirit Action Expansion Design

## Background

The companion spirit currently has a small action set in `xy_pet_spirit_lighthouse.glb`, generated by `blender/xy_pet_spirit_actions.py`: `FeedTreat`, `TalkListen`, `BondGlow`, `SleepFloat`, and `SecretTwirl`.

The frontend already routes companion interactions through `CompanionAnimation` in `frontend/src/lib/companionSpirit.ts` and triggers animations from `frontend/src/components/ExploreMode.tsx`. The user approved expanding all three visual directions: daily affection, exploration reactions, and emotional care.

## Goal

Add a balanced six-action package that makes the spirit feel more alive without turning the model pipeline into a large animation project.

## Action Package

| Category | Clip | Purpose | Primary Trigger |
|---|---|---|---|
| Daily affection | `Nuzzle` | The spirit softly rubs close and tail-wags. | Petting the spirit or opening the panel after idle time. |
| Daily affection | `CuriousPeek` | The spirit leans forward, ear fins tilt, orb drifts closer. | Player starts talking or types a short message. |
| Exploration reaction | `DiscoveryHop` | A quick hop with sparkle-like body/orb motion. | Discoveries, collection, fish catch, nearby surprise. |
| Exploration reaction | `LanternGaze` | The spirit rises and tilts upward as if watching lanterns. | Single or flock lantern release. |
| Emotional care | `ComfortPulse` | Slow bow and warm pulse around halo/orb parts. | Worried, anxious, sad, tired, or helpless emotion response. |
| Emotional care | `NightGuard` | Slow vigilant float with stronger lighthouse/halo glow. | Deep night companion opening or night exploration. |

## Architecture

The implementation stays inside the existing companion pipeline:

| Layer | Change |
|---|---|
| Blender script | Extend `CLIPS` and add procedural keyframes in `blender/xy_pet_spirit_actions.py`. |
| Model asset | Re-export `frontend/public/models/xy_pet_spirit_lighthouse.glb` with the new NLA tracks. |
| Frontend type | Add the six clip names to `CompanionAnimation`. |
| Frontend routing | Map existing interaction outcomes to more specific clips before calling `triggerCompanionAction`. |
| Tests | Extend companion animation normalization / trigger tests so unknown names still fall back safely and new names are accepted. |

NLA track means a named reusable animation clip embedded in the GLB model. The frontend can request the clip by name without knowing the keyframe details.

## Data Flow

1. A player action happens, such as petting, talking, discovering, fishing, releasing lanterns, or opening the spirit panel at night.
2. `ExploreMode.tsx` resolves the user-facing reply and chooses a `CompanionAnimation`.
3. The companion model receives `{ name, nonce }`; `nonce` forces repeated requests for the same clip to replay.
4. The Three.js companion component plays the matching GLB clip if available.
5. If a clip is missing or unknown, existing fallback behavior keeps the spirit usable.

## Trigger Rules

| Scenario | New Clip |
|---|---|
| Petting the spirit | `Nuzzle` |
| Empty talk / beginning to listen | `CuriousPeek` or `TalkListen`, depending on emotion |
| Discover, collect, fish catch, nearby surprise | `DiscoveryHop` |
| Lantern release / lantern flock | `LanternGaze` |
| Worried emotional response | `ComfortPulse` |
| Deep night opening / night visit secret | `NightGuard` |

Existing actions remain valid. `SecretTwirl` still overrides normal clips when a secret unlocks, because unlock feedback should stay visually special.

## Error Handling

| Risk | Handling |
|---|---|
| A clip name is requested before the GLB contains it. | Normalization accepts known names, but model playback should fall back to idle/available clips rather than breaking render. |
| Blender node names differ between model revisions. | `keyed_action` already no-ops when a target object is missing; the script should keep using optional targets and animate the root as the minimum visible behavior. |
| Re-export changes unrelated model data. | Keep the Blender script minimal and verify exported clip names rather than manually editing the GLB. |
| Too many energetic clips make the spirit noisy. | Use calm timings for care clips; keep `DiscoveryHop` short and event-triggered. |

## Testing

| Test | Expected Result |
|---|---|
| Unit test for animation normalization | The six new clip names return themselves; invalid names fall back to a safe existing action. |
| Unit test for companion interactions | Pet/talk/care paths choose the expected new action where deterministic. |
| Model inspection script or GLB parser check | Exported GLB contains all six new animation names. |
| Frontend build | TypeScript accepts the expanded `CompanionAnimation` union. |
| Manual browser check | Petting, releasing lanterns, and night opening visibly trigger distinct spirit motions. |

## Scope Boundaries

This change does not add new UI controls, new food, new secrets, new backend endpoints, or a full animation selector. It only expands the companion spirit's reusable action library and maps existing events to richer motions.

## Acceptance Criteria

- `xy_pet_spirit_lighthouse.glb` includes `Nuzzle`, `CuriousPeek`, `DiscoveryHop`, `LanternGaze`, `ComfortPulse`, and `NightGuard`.
- Frontend types and normalization accept the six new names.
- Existing companion interactions continue to work.
- Secret unlocks still use `SecretTwirl`.
- Automated tests cover new action names and at least the deterministic trigger paths.
