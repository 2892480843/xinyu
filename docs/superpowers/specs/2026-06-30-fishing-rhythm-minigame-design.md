# 拾海垂钓节奏收线小游戏设计

## 背景

当前项目是 Vite + React + Three.js 的前端应用，`frontend/src/components/ExploreMode.tsx` 承担「上岛走走」自由探索体验。垂钓功能已经存在基础链路：玩家走到海湾岸边后出现「垂钓」按钮，点击后显示 `xy_item_fishing_bobber.glb` 浮标，等待随机时间进入咬钩状态，再点击「收线」获得贝壳、漂流瓶、小鱼或星海鱼，并通过 `xy_catch` 写入本地计数。

用户已明确选择「完整小游戏」方向，并确认核心判定采用「节奏收线」：鱼咬钩后显示收缩光环，玩家需要在正确窗口内点击或按键收线。

## 目标

- 将现有「等待后点一下收线」升级为可感知、可失败、可成功的节奏小游戏。
- 保留当前海湾触发、浮标 GLB、收获卡、`xy_catch` 计数和精灵陪聊事件。
- 支持桌面端点击/键盘与移动端触控，不新增复杂手势。
- 让失败反馈温和短暂，不用惩罚式弹窗打断治愈氛围。
- 将随机与判定逻辑集中到小型纯函数或局部工具中，便于测试。
- 控制实现范围，不引入后端持久化、鱼类图鉴、任务成就或新大型依赖。

## 非目标

- 不做力度条、方向追踪、多段连击或鱼挣扎张力系统。
- 不新增后端 API，不改变记忆、岛屿状态、情绪分析或陪伴精灵接口。
- 不替换当前浮标 GLB，也不要求新增鱼竿、鱼线或鱼类 GLB。
- 不把垂钓拆成独立页面；它仍是 `ExploreMode` 海湾内的轻小游戏。
- 不改变其它自由探索玩法，如种花、天灯、风铃、开车和海螺互动。

## 方案选择

| 方案 | 核心思路 | 优点 | 风险/成本 | 结论 |
|---|---|---|---|---|
| 单击节奏窗口 | 咬钩后光环持续收缩，在高亮命中窗口内收线即成功 | 移动端友好，节奏明确，最贴现有治愈体验 | 判定参数需要调得宽松，避免挫败 | 采用 |
| 多段节奏 | 连续命中 2-3 次才完成收线 | 游戏性更强 | 容易打断散步节奏，移动端误触成本更高 | 不采用 |
| 随机事件版 | 在原有点击收线后仅增加结果文案和概率事件 | 改动小 | 不符合用户选择的「完整小游戏」目标 | 不采用 |

## 玩法流程

| 阶段 | UI 表现 | 玩家操作 | 成功/失败条件 |
|---|---|---|---|
| `idle` 空闲 | 海湾岸边出现「垂钓」按钮 | 点击按钮 | 进入 `cast` |
| `cast` 抛竿 | 按钮显示「抛竿中...」，水面显示浮标 | 无需操作 | 短暂过渡后进入 `waiting` |
| `waiting` 等鱼 | 按钮显示「等鱼靠近...」，浮标轻微上下漂 | 可点击取消或离开岸边取消 | 随机等待后进入 `bite` |
| `bite` 咬钩判定 | 底部出现收缩光环、命中窗口和「收线」按钮 | 点击按钮、空格或回车 | 当前节奏值落入命中窗口则成功，否则失败 |
| 成功结果 | 弹出原有收获卡，显示本次收获与累计数 | 点击「收下」 | `FishingState` 回到 `idle`，收获卡由 `shownCatch` 控制 |
| `missed` 失败 | 底部短提示「鱼从光里游走了」 | 等提示消失或再次垂钓 | 回到 `idle` |

状态机（State Machine，把玩法拆成固定阶段）建议从当前三态扩展为：

```text
idle -> cast -> waiting -> bite -> idle + shownCatch
                         \-> missed -> idle
```

离开海湾、水边条件失效、进入 NPC 对话或卸载组件时，所有计时器都应清理，状态回到 `idle`。

## 节奏判定

咬钩阶段维护一个 `rhythmStartedAt` 时间戳，渲染层根据当前时间计算光环收缩进度：

| 参数 | 建议值 | 说明 |
|---|---|---|
| 总窗口时长 | `1800ms` | 从咬钩到鱼跑走的总时间 |
| 命中窗口 | `0.38 - 0.62` | 进度落入该区间时点击成功 |
| 早点击 | `< 0.38` | 显示「别急，它刚碰到浮标」 |
| 晚点击 | `> 0.62` | 显示「鱼从光里游走了」 |
| 自动错过 | `>= 1` | 未操作自动失败 |

节奏进度是 `clamp((now - rhythmStartedAt) / duration, 0, 1)`。命中判断由一个小函数承载，例如 `isFishingRhythmHit(progress, window)`，便于自动测试覆盖边界。

## UI 设计

底部 HUD（屏幕操作提示）继续固定在现有垂钓按钮位置，避免和其它自由探索按钮争抢层级：

- `idle`：圆角按钮「🎣 垂钓」。
- `cast`：按钮或提示「抛竿中...」，不可重复触发。
- `waiting`：提示「等鱼靠近...」，可保留轻微脉冲。
- `bite`：显示一个紧凑小游戏面板，包含收缩光环、目标环、状态文案和「收线」按钮。
- `missed`：显示一条短暂玻璃提示，不弹大框。
- `caught`：沿用当前收获卡。

光环实现优先使用普通 HTML/CSS（层叠样式表，负责页面样式）覆盖层，而不是在 Three.js 场景里新增复杂对象。理由是移动端触控和响应式布局更可控，也便于测试检查关键文案和 class。

## 数据与概率

第一版继续使用本地数据：

| 数据 | 存储 | 说明 |
|---|---|---|
| 累计收获数 | `localStorage.xy_catch` | 沿用现有字段 |
| 本次收获 | React state | 弹卡展示后即清除 |
| 等鱼随机时间 | 内存计时器 | 每次抛竿生成，离开海湾清理 |
| 收获类型 | 内存随机 | 沿用现有 `CATCHES` 列表 |

若要让测试稳定，收获随机可先保持现状，但节奏判定逻辑必须可用纯函数测试。未来需要图鉴时，再把 `CATCHES` 抽到独立 `lib/fishing.ts` 并增加 `id`、稀有度和历史记录。

## 组件与代码边界

优先小步改造 `ExploreMode.tsx`，避免大规模重构：

- 新增 `FishingState` 类型：`idle | cast | waiting | bite | missed`；成功后仍通过 `shownCatch` 控制收获卡，不必新增 `caught` 独立状态。
- 新增节奏参数常量，如 `FISHING_RHYTHM_DURATION_MS`、`FISHING_HIT_START`、`FISHING_HIT_END`。
- 新增纯函数：`fishingRhythmProgress(now, startedAt, duration)` 和 `isFishingRhythmHit(progress)`。
- 新增 `FishingRhythmHud` 小组件，负责渲染咬钩阶段光环和按钮。
- 保留 `FishingSpot` 与 `GltfFishingBobber`，只把 `casting` 条件调整为非 `idle` 时可见。

如果 `ExploreMode.tsx` 继续膨胀影响可读性，后续迭代再将纯函数抽入 `frontend/src/lib/fishing.ts`。本轮优先最小改动闭环。

## 错误处理与边界

| 场景 | 处理 |
|---|---|
| 玩家离开海湾 | 清理计时器、隐藏浮标、回到 `idle` |
| 玩家在 `cast` 或 `waiting` 重复点击 | 不重复设置计时器 |
| 玩家过早收线 | 进入 `missed`，显示轻提示后回到 `idle` |
| 玩家过晚收线 | 进入 `missed`，播放水波或轻提示 |
| 玩家不操作 | 咬钩窗口结束后自动 `missed` |
| 收获卡打开时继续点背景 | 背景点击关闭，现有行为保留 |
| 本地存储不可用 | 捕获异常，玩法仍能进行，只是不保存累计数 |

## 测试与验证

| 验证项 | 标准 |
|---|---|
| 自动测试 | 新增或扩展 `frontend/tests/fishingBobberModel.test.mjs`，检查节奏状态、命中窗口常量、HUD 文案、错过反馈和浮标 GLB 仍被使用 |
| 构建 | 在 `frontend` 下执行 `npm run build` 通过 |
| 全量测试 | 在 `frontend` 下执行 `npm test` 通过 |
| 视觉手测 | 进入上岛探索，移动到海湾，点击垂钓，确认浮标出现、光环收缩、命中弹收获卡、错过显示短提示 |
| 移动端手测 | 确认底部 HUD 不遮挡系统安全区，按钮文字不溢出 |

实现时采用 TDD（测试驱动开发，先写会失败的测试，再写最小代码让测试通过）：先补覆盖节奏小游戏的红测，再改 `ExploreMode.tsx`，最后跑测试与构建。

## 验收标准

- 在海湾岸边可以启动垂钓小游戏。
- 抛竿后浮标可见，等待期间不会重复触发多个计时器。
- 咬钩后显示收缩光环和明确的「收线」操作。
- 在命中窗口内点击可以获得收获卡，累计 `xy_catch` 增加。
- 过早、过晚或不操作都会出现温和失败提示，并能再次垂钓。
- 离开海湾会取消当前垂钓状态。
- 桌面点击、空格或回车，以及移动端触控都能完成收线。
- 现有种花、天灯、风铃、海螺和 NPC 入口不因垂钓 HUD 失效。
- `npm test` 与 `npm run build` 通过。

## 风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| 命中窗口太窄 | 玩法显得挫败 | 第一版窗口设为 24% 总进度，并用文案鼓励重试 |
| `ExploreMode.tsx` 继续变长 | 后续维护变难 | 本轮只新增小函数和小组件；若后续加图鉴再拆 `lib/fishing.ts` |
| 多计时器竞争 | 状态跳转错乱 | 所有 `setTimeout` 由状态 `useEffect` 返回清理函数 |
| HUD 与其它按钮重叠 | 移动端误触或遮挡 | 复用现有底部定位，并继续受 `atWater`、`nearNpc` 条件约束 |
| 随机测试不稳定 | 自动测试偶发失败 | 测试聚焦纯判定函数和源码结构，不依赖随机收获 |

## 术语说明

| 术语 | 说明 |
|---|---|
| State Machine | 状态机，把玩法拆成固定阶段，避免多个布尔值互相冲突 |
| HUD | Head-Up Display，游戏界面上的操作提示和状态面板 |
| Three.js | Web 3D 渲染库，用于绘制岛屿、模型、光照、天空和相机 |
| GLB | 3D 模型文件格式，当前垂钓浮标是 `xy_item_fishing_bobber.glb` |
| React state | React 组件内部状态，用来驱动界面显示和交互阶段 |
| `localStorage` | 浏览器本地存储，用来保存累计垂钓收获数 |
| TDD | 测试驱动开发，先写失败测试，再写功能代码，最后重构保持测试通过 |
