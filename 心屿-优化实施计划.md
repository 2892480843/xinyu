# 《心屿》优化实施计划 v1

> **目标**：把当前"旧版线性管道"升级为需求文档 v3.0 描述的"AI 驱动情绪世界生成游戏"。
> **核心改造**：① 心象岛屿生长系统 ② 多智能体导演台 ③ 心象地图 ④ 演示稳定性。
> **编写日期**：2026-06-16　**状态**：✅ 全部 6 个 Phase 已实现并验证通过（后端 10 项单测 + 前端 build/lint + WS 端到端流式）。

## Phase 1：心象岛屿生长系统（核心创新①，1.5 天）

> 这是"chatbot → 游戏"的分界线。纯 Python 计算，**不依赖 LLM**，保证演示稳定。

**新增 `backend/app/services/island_state_service.py`**

```
compute(current_emotion, current_intensity, history) -> IslandState
```

| 字段 | 计算逻辑 |
|---|---|
| `growth_level` | 按记忆总数：1(0-1) / 2(2-4) / 3(5-9) / 4(10-19) / 5(20+) |
| `dominant_emotion` | 最近 10 条记忆里出现最多的情绪 |
| `trend` | recovering(负面强度下降) / brightening(正面增多) / stormy(连续高强度负面) / stable(波动小) / mixed(多情绪交替) |
| `features[]` | 按 dominant_emotion 查"岛屿元素映射表"（需求文档 6.3.4），随 growth_level 累积元素数量 |
| `weather_memory` | recent_rain / misty / clearing_sky，由近期情绪推导 |
| `summary` | 模板句拼装（可选：交给 Narrative Agent 用 LLM 润色一句） |

**接入 `backend/app/main.py`**
- 在 scene 之后、保存记忆之前调用，写入 `ReflectResponse.island_state`
- WebSocket 增加 `island_state` 事件推送（在 scene 与 narrative 之间）

**可选接口**：`GET /api/island-state?user_id=` —— 用户首次进入未提交也能看到当前岛屿（需求文档 9.5）。**建议做**，回访体验关键。

**前端展示**
- `frontend/src/components/NarrativeCard.tsx`：叙事卡片下方加"岛屿状态摘要"区（显示 `island_state.summary` + growth_level 等级 + features 图标）
- `frontend/src/components/IslandScene.tsx`：根据 `features[]` 叠加可见元素（灯塔/花/星光）——MVP 先做 3 类可见变化即可满足验收

**验收**：记忆数变化→growth_level 变化；不同主导情绪→不同 features；前端至少 3 类可见岛屿变化。

## Phase 2：多智能体导演台（核心创新②，1 天）

> 后端五个阶段已存在，本质是**重构 + 展示层**，投入最低、对评委收益最高。

**后端重构 `backend/app/main.py`**
- 把 L58-99 的线性流程抽成一个生成器/流程函数，每个阶段产出一条 `AgentTraceItem`：

| Agent | label | output 示例 |
|---|---|---|
| emotion | 情绪分析 Agent | 识别为疲惫，强度 0.72 |
| memory | 记忆检索 Agent | 找到 3 条最近记忆，其中 2 条与疲惫有关 |
| environment | 环境推理 Agent | 夜色 + 星光场景，灯塔已点亮 |
| narrative | 叙事表达 Agent | 已生成 50-120 字治愈叙事 |
| safety | 安全边界 Agent | 未触发 / 已触发安全边界 |

- HTTP：收集全部 trace 放进 `ReflectResponse.agent_trace`
- WebSocket：每个 Agent 完成时推 `{event:"agent", agent, label, status:"done", output}`（复用现有 0.12s 间隔节奏）

> ⚠️ **诚实性**：保持"可解释 AI 流水线"话术，或让 environment/narrative 真正各自带独立 prompt 做 LLM 推理；不空喊 autonomous multi-agent，避免被评委戳穿。

**前端新增 `frontend/src/components/AgentDirectorPanel.tsx`**
- 5 张 Agent 卡片：名称 + 状态灯（等待/处理中/已完成）+ output 摘要
- 消费 WS `agent` 事件逐个点亮；HTTP 回退时一次性渲染 `agent_trace`
- 在 `Home.tsx` 的 loading 阶段展示

**验收**：提交后导演台按阶段点亮，每个 Agent 有可读摘要；高风险时 Safety Agent 显示已触发；HTTP 回退也能渲染。

## Phase 3：后端补漏（可信度，1 天）

并行于前端开发，修掉评委会戳的硬伤。

1. **imprint 入库**　`memory_service.py` L29-44：表加 `imprint TEXT DEFAULT ''` 列 + 迁移逻辑；`main.py` 保存时写入；`/api/memories` 返回。（否则 v3.0 的"印记收藏墙"无地基）
2. **安全关键词兜底**　`safety_service.py`：在阈值之外加自伤/绝望关键词黑名单（"不想活""撑不下去""结束自己"等），命中即触发——**修复"平静措辞自伤因强度低而漏过"的真实安全漏洞**。
3. **Mock 去重复**　`llm_provider.py` L70-151：模板选取引入轻随机/轮换，避免同输入永远同输出在现场露馅。
4. **接真实模型预案**　演示用腾讯云大模型 / OpenAI 兼容接口（`LLM_PROVIDER=openai` + `OPENAI_BASE_URL`），keyword-mock 仅作降级。提前配好 `.env` 并验证。

**验收**：imprint 可存可查；关键词样例稳定触发安全；连续相同输入叙事有变化。
